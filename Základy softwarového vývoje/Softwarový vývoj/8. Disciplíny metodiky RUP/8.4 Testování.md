**Testování** a **Zajištění kvality (Quality Assurance – QA)** jsou klíčové disciplíny v procesu vývoje softwaru.

## 1. Základní pojmy

| Pojem | Popis |
| :--- | :--- |
| **Zajištění kvality (QA)** | Širší pojem zahrnující **všechny činnosti** prováděné s cílem dosažení **požadované kvality** produktu. Zahrnuje i testování. |
| **Testování** | Jedna z možných metod v procesu QA. Je to **jedna z kontrol kvality**. Definuje se jako proces **získání důvěry** v program nebo **ověřování programu za účelem hledání chyb**. |
| **Dostatečná kvalita** | Software je "dostatečně kvalitní", pokud je **použitelný** a splňuje následující: **Použitelnost a užitečnost** by měla značně převažovat nad zbylými problémy (málo chyb, nepatrné chyby, které nebrání používání). **Nesmí** obsahovat žádné **kritické chyby**. |
| **Softwarová chyba (bug)** | Software **nedělá něco, co by dle specifikace měl dělat**, nebo **dělá něco, co by neměl**. Software dělá něco, o čem se specifikace **nezmiňuje**. |

### Verifikace a Validace

Jedná se o dva rozdílné, ale důležité pojmy, které se nesmí zaměňovat:

1.  **Validace (Validation – "Do right things"):** Proces ověřování, **zda se vyvíjí správná specifikace**, tj. **zda děláme to, co zákazník požaduje**. Není to testování! Zabývá se jím disciplína **Requirements management** (Správa požadavků) a probíhá **intenzivně v počátečních fázích**.
2.  **Verifikace (Verification – "Do things right"):** Proces ověřování **shody produktu se specifikací**, tj. **zda je produkt implementován v souladu s požadavky**. Jedná se o **proces testování**.

## 2. Fáze a Role v Testování

Testování je třeba nastartovat co nejdříve a provádět **průběžně**. Testování není záležitostí pouze testovacího týmu; do procesu jsou zapojeny další role:

| Role v testování | Stručný popis odpovědnosti |
| :--- | :--- |
| **Test Manager** | Řídí celý proces testování, schvaluje požadavky na testování, plánuje testování, přiděluje zdroje (HW, SW, HR, Tools). |
| **Test Analyst** | Analyzuje požadavky na testování, identifikuje předměty a **navrhuje strategii** testování (např. objekty, Use Cases). |
| **Test Designer** | Navrhuje **detailní podobu testování** a jeho potřebu (např. Test Cases, Test Scripts, Test Procedures, Test Data). Sestavuje **Master Test Set** a **Iteration Test Set**. |
| **Test Implementer** | Implementuje potřebné **testovací skripty** a připravuje příslušná testovací data. |
| **Test Deployment Executive** | Řídí a kontroluje **jednotlivé úkoly při implementaci testů** a ověřuje jejich správnost. |
| **Test Auditor** | Kontroluje **správnost testovacích postupů** na vyšší úrovni (dodržování metodik, zásad, standardů). |
| **Product Auditor** | Provádí **testy dle Test Scripts** v iteracích, zaznamenává výsledky a chyby. |

## 3. Typy testů (dle modelu FURPS+)

Typologie testů v RUP odpovídá rozměrům kvality modelu **FURPS+** (Funkčnost, Použitelnost, Spolehlivost, Výkonnost, Podpora a Bezpečnost).

### A. Dělení dle provedení a rozsahu (Kaskádový model testování)

| Typ testu | Kdo provádí | Účel a Popis |
| :--- | :--- | :--- |
| **Unit test** | Vývojový tým | První a důležitá fáze. Vývojář testuje **dokončené části programu** (unit, use casy, komponenty). |
| **Integration test** | Vývojový tým | Ověření, že **dodané části** (unity, komponenty) lze **vzájemně integrovat** a vytvořit funkční celek (build). |
| **Smoke test** | Testovací tým | Rychlé ověření **stability dodaného buildu**. Provádí se na začátku testovacího cyklu. Pokud build neprojde, **nelze pokračovat** v hlubokovém testování. |
| **System test** | Testovací tým | **Hlubokový test** celého systému. Zahrnuje **téměř všechny typy testů (FURPS)**. Důležitý je **regresní test**, který ověřuje, zda nové funkce **neporušily funkčnost** stávajících. |
| **Acceptance test (UAT)** | Zákazník | Testování na **vlastním testovacím/provozním prostředí** zákazníka. Nejvyšší forma ověření shody produktu s požadavky. |

### B. Dělení dle FURPS+

| Kategorie | Příklady testů |
| :--- | :--- |
| **Funkčnost** | **Function test** (ověření funkčních požadavků uvedených v návrhu systému). |
| **Použitelnost** | Testy uživatelského rozhraní (GUI), on-line nápověd, školicích materiálů, dokumentace. |
| **Spolehlivost** | **Structure test** (chybné odkazy, nefunkční stránky), **Integrity test** (odolnost proti selhání), **Volume test** (stabilita při velkém objemu dat). |
| **Výkonnost** | **Performance test** (měření odezvy), **Load test** (chování systému pod vysokou zátěží), **Stress test** (při nestandardních podmínkách), **Benchmark test** (porovnání s referenčním prostředím). |
| **Podpora** | **Configuration test** (funkčnost na různých HW/SW konfiguracích), **Installation test** (korektní instalace a aktualizace). |
| **Bezpečnost** | **Security test** (ověření přístupových práv), **Vulnerability scan** (zranitelnost systému), **Penetration test** (neoprávněný přístup), **Resiliency test** (odolnost proti DDoS). |
